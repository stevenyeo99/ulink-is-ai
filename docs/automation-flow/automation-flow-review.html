JD1 and JD2 Automation Flow Review

This review pack consolidates the current automation-flow documents under docs/automation-flow into one file for stakeholder review. It is intended to confirm the proposed ownership split, handover boundary, and main automation control points before implementation.

JD1 Scope Intake, OCR, pre-assessment, Console upload, and handover to JD2.
JD2 Scope Claim preparation, IAS create or revise logic, and final status handling.
Handover Rule JD2 starts only after JD1 marks the case ready and upload is verified.

1. Review Objective

The target operating model is to keep JD1 and JD2 as related but distinct automation stages. JD1 is responsible for turning an incoming case into a validated, uploaded, JD2-ready package. JD2 is responsible for turning that package into a valid IAS claim submission or provider-claim revision.

Core design decision: OCR and intake stay in JD1. Claim creation and amendment stay in JD2.

2. Scope Split

JD1 Responsibilities

  • Poll eligible Freshdesk or email cases for JD1 processing.
  • Download attachments and retrieve large files when needed.
  • Normalize files and run OCR plus first-pass extraction.
  • Check document completeness and basic readiness.
  • Send missing-document requests or escalate exception cases.
  • Upload validated documents into Console via VM SFTP or API.
  • Create the JD2-ready handover package and update status.

JD2 Responsibilities

  • Consume only JD1-approved and JD1-uploaded cases.
  • Validate handover data for claim-processing purposes.
  • Prepare IAS claim fields, fee lines, and coding inputs.
  • Check coverage, NOC, and claim-processing rules.
  • Run create-new-claim or revise-existing-claim logic.
  • Update IAS statuses and Freshdesk audit notes.
  • Mark the JD2 case completed or route to manual review.

3. End-to-End View

High-Level Automation Sequence
JD1 intake and attachment retrieval -> OCR and pre-assessment
If incomplete or invalid: missing-document workflow or escalation
If valid: upload documents to Console and verify success
Create JD2-ready handover package -> JD2 consumes approved case
JD2 validates claim-processing requirements and builds IAS claim payload
If no existing provider claim number: create new IAS claim
If existing trusted provider claim number: revise existing IAS claim
Update IAS status and Freshdesk note -> complete or manual review

4. JD1 Flow Summary

JD1 is the intake and pre-assessment stage. It should determine whether a case is complete enough to move forward, and it should upload validated documents into Console before handover.

Stage Expected JD1 Outcome
Case selection Only eligible JD1 tickets are picked. Duplicate or manual-only cases are skipped.
Attachment handling Attachments are downloaded, normalized, and made available for OCR.

Important JD1 boundary: JD1 does not prepare claim lines, apply claim coding, or create or revise the IAS claim record.

5. JD2 Flow Summary

JD2 starts only after JD1 handover is complete. JD2 should not redo the raw intake workflow. Its job is to turn the approved case package into a valid IAS transaction.

Stage Expected JD2 Outcome
Handover intake JD2 loads the JD1 package, including extracted fields, normalized documents, and Console references.
Claim-entry validation JD2 validates treatment dates, diagnosis, identifiers, coverage, and NOC requirements.
Claim building JD2 builds line-level claim items and applies claim-specific rules and exclusions.
IAS processing JD2 either creates a new IAS claim or revises an existing provider claim.
Completion JD2 verifies final identifiers, updates statuses, and records audit notes.

6. JD1 to JD2 Handover Contract

JD2 should receive a structured handover package rather than reprocessing the raw email thread from scratch. The minimum fields currently proposed are below.

Field Purpose
freshdesk_ticket_id Primary case reference across JD1 and JD2.
case_type Supported flow classification such as provider claim or reimbursement claim.
jd1_status Expected to indicate that the case is ready for JD2.
handover_version Supports reprocessing when a case changes and helps prevent duplicates.
source_email_metadata Retains sender, subject, thread, and intake context.
normalized_document_paths References to the validated working copies prepared by JD1.
ocr_result OCR JSON or reference to OCR output generated by JD1.
extracted_fields First-pass document fields used by JD2 as primary input.
document_completeness_result Indicates what JD1 checked and whether the case passed pre-assessment.
console_upload_refs Uploaded file identifiers or references needed for downstream traceability.
existing_claim_no Signals whether JD2 should consider provider-claim revision logic.

7. Control Points and Risk Gates

Control Points

  • Prevent duplicate processing across cron runs.
  • Block JD2 until JD1 upload is verified.
  • Version the handover when new files or new data arrive.
  • Keep create-versus-revise logic explicit in JD2.

Risk Gates

  • Missing or unsupported attachments.
  • Pre-assessment mismatch or escalation case.
  • Console upload failure.
  • Coverage, NOC, or policy issue in JD2.
  • IAS submission or amendment failure.

8. Suggested Statuses

  • JD1 - Pending Documents
  • JD1 - Escalation
  • JD1 - Upload Exception
  • JD1 - Ready for JD2
  • JD2 - Processing
  • JD2 - Manual Review
  • JD2 - Completed
